# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

This is a monorepo for managing vendor integration packages under the @zerobias-org organization. Each vendor has its own package in the `package/` directory with standardized structure including metadata, logos, and version management.

## Essential Commands

### Development Setup
```bash
# Install root dependencies (commitlint hooks, tsx for scripts/correctDeps.ts)
npm install
```

All 464 vendors are on the gradle pipeline as of May 2026. No lerna/nx infrastructure remains — see `chore(repo-cleanup)` history for the removal.

### Working with Vendors
```bash
# Create a new vendor package: copy template + add gradle marker
./scripts/createNewProduct.sh package/[vendor-name]
echo 'plugins { id("zb.content") }' > package/[vendor-name]/build.gradle.kts
# Then fill in package.json, index.yml, and add an appropriate logo

# Validate via gradle (per-package validator + dataloader)
./gradlew :[vendor-name]:gate

# List all auto-discovered vendor projects
./gradlew projectPaths

# Cross-cut: ensure no two vendors share an id UUID
./gradlew validateUniqueIds

# Fix dependency declarations (per-package helper)
cd package/[vendor-name] && npm run correct:deps
```

### Publishing and Version Management
Publishing is driven by the gradle `Publish` GitHub Actions workflow,
triggered on push to `main` / `qa` / `dev` / `uat`. It:
1. `detect` — diffs `package/**/build.gradle.kts` against `origin/main` to find changed vendors
2. matrix `publish (<vendor>)` — runs `zbb publish` per vendor (gate preflight via committed `gate-stamp.json` → `npm publish --tag next` → cumulative `promoteAll` to dev/qa/uat/latest)
3. `sync` — propagates main → uat → qa → dev after a successful main publish

```bash
# Manually trigger the workflow on a branch (publishes ALL gradle-detected vendors)
gh workflow run publish.yml --repo zerobias-org/vendor --ref dev

# Local dry-run of the publish flow for one vendor
./gradlew :[vendor-name]:publish -PdryRun=true
```

## Architecture

### Package Structure
Each vendor package (`package/[vendor-name]/`) contains:
- `package.json` - NPM package metadata (uses `zerobias:` config block)
- `index.yml` - Vendor configuration and metadata
- `logo.{svg|png|jpg}` - Vendor branding (any one extension)
- `build.gradle.kts` - **REQUIRED** one-line marker `plugins { id("zb.content") }`. Without it, the `Publish` workflow's `detect` job won't pick the vendor up and it will never reach the registry.
- `.npmrc` - Registry config
- `gate-stamp.json` - Generated by `./gradlew :<vendor>:gate`, committed to record validate + dataloader passed
- `CHANGELOG.md` - Version history (auto-generated by gradle's release flow)

### Key Technologies
- **Gradle (`zb.content` plugin)**: Drives validate / gate / publishNpm / promoteAll per vendor. Plugin lives in `zerobias-org/util/packages/build-tools` and resolves via `settings.gradle.kts`.
- **`zbb`**: Lifecycle CLI used by CI; `zbb publish` translates to gradle tasks per `zbb.yaml`.
- **`zbb-publish-reusable.yml`**: Shared GitHub Actions workflow that runs `detect` → `version` (single-writer on main) → matrix `publish` → `update-bundle` → `sync`.
- **Validator philosophy**: The dataloader is the source of truth for schema (UUID format, code regex, status enum, URL parse, tag UUIDs, etc.). The inline `extra["contentValidator"]` in `build.gradle.kts` only enforces things the dataloader **cannot** check: (1) filesystem ↔ npm ↔ `zerobias.package` triangulation via `SchemaPrimitives.requirePackageIdentity`, (2) logo file correctness when present — magic bytes, size, files-array reference, `<svg>`-content check for SVGs, (3) repo-wide unique `id` UUIDs (`:validateUniqueIds` root task). The `testIntegrationDataloader` step during `:gate` runs the canonical dataloader check against an ephemeral Neon Postgres branch.
- **TypeScript / tsx**: Used by the per-package `correct:deps` helper (`tsx ../../scripts/correctDeps.ts`).
- **Conventional Commits**: Enforced via commitlint.

### Publishing Flow
Driven by the `Publish` GitHub Actions workflow on push to `main` / `qa` / `dev` / `uat`. Pipeline:
1. `detect` — diffs `package/**/build.gradle.kts` against `event.before` to find changed vendors.
2. `version` (main only) — single-writer pre-matrix; `zbb version` aggregates per-package bumps into one commit.
3. matrix `publish (<vendor>)` — runs `zbb publish` per vendor (`publishGuard` checks the committed `gate-stamp.json` → `npm publish --tag next` → `promoteAll` to dev/qa/uat/latest).
4. `update-bundle` — refreshes `@zerobias-org/vendor-bundle` deps from npm and publishes the next patch.
5. `sync` — propagates `main → uat → qa → dev` after success.

## Important Notes

- **Authentication**: Set `ZB_TOKEN` environment variable for npm registry access
- **Commit Format**: All commits must follow Conventional Commits specification
- **Private Registry**: Packages publish to `npm.pkg.github.com/@zerobias-org`
- **No Direct npm publish**: Publishing is driven by the gradle `Publish` workflow on push to `main`/`qa`/`dev`/`uat`; locally use `./gradlew :<vendor>:gate` to validate before push.
- **Vendor Naming**: Package names must match `@zerobias-org/vendor-{code}` where `{code}` matches `^[\d_a-z]+$` (lowercase alphanumeric with underscores allowed) — same regex enforced by `com/platform/dataloader` `VendorFileHandler` and by this repo's gradle validator. The UI's `vspCodeValidator` is stricter (no underscores), so prefer plain lowercase alphanumeric for new codes.
